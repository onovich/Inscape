# DSL 语言设计草案

状态：草案，语法未定稿

## 设计原则

- 优先保持阅读连续性，避免让普通对白充满控制符。
- 普通文本应该容易写，复杂逻辑应该清楚地显式表达。
- 解析结果必须确定，同一份脚本在同一编译器版本下产生稳定 IR。
- 语法错误要能定位到行和列，并尽量给出修复建议。
- 文本、标签、变量和分支都要能映射到源位置和锚点。

## 已有方向

### 自然对话

目标是支持接近自然剧本的写法：

```inscape
明里：你真的要进去吗？
悠：如果现在停下，就永远不会知道里面有什么。
```

待确认：

- 角色名和对白之间使用全角冒号、半角冒号，还是两者都支持。
- 旁白是否允许无角色名直接书写。
- 多行对白如何表达。
- 角色别名、本地化显示名、语音资源如何绑定。

### 行内标签

目标是让演出指令嵌入文本流：

```inscape
[bg: classroom]
[show: akari happy at center]
明里：今天的教室好安静。
```

待确认：

- 标签语法是否统一为 `[command: args]`。
- 标签是否允许多个参数、命名参数和字符串。
- 标签是否可以出现在对白行内部，还是只能独立成行。
- 标签失败时是编译错误、运行时错误，还是可降级警告。

### 条件与变量

目标是支持简单叙事逻辑：

```inscape
if score > 10
  明里：你居然记得。
else
  明里：算了，当我没问。
end
```

待确认：

- 缩进是否有语义。
- 条件块是否必须以 `end` 结束。
- 变量类型是否限定为 bool、number、string。
- 是否支持列表、字典、枚举或标签集合。
- 是否允许表达式中调用函数。

## 候选语法元素

- `@label` 或 `#label`：显式节点标签，用于跳转和编辑器大纲。
- `choice`：玩家选项。
- `set`：变量赋值。
- `jump`：跳转到显式节点。
- `call` / `return`：子场景调用，是否需要待确认。
- `include`：跨文件引用，是否需要待确认。
- `//`：注释。是否支持块注释待确认。

## 暂不确定的设计边界

- 是否允许非线性图式结构，还是保持单文件单流为主。
- 分支合流是否需要显式语法。
- 是否允许作者手写稳定 ID，作为隐式哈希的补充。
- 是否支持宏、模板或自定义语法扩展。
- 是否允许在 DSL 中直接写复杂演出时间轴。

## 下一步建议

1. 选取一个 3 到 5 分钟视觉小说片段作为语法样本。
2. 用两到三种语法候选重写同一片段，比较阅读负担与解析复杂度。
3. 明确“最小可用语法”：对白、旁白、标签、变量、条件、选择、跳转。
4. 决定缩进、块结束符、注释、字符串和转义规则。
5. 再选择 Antlr4 或 Superpower，并开始解析原型。
