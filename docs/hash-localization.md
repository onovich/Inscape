# 哈希锚点与本地化

状态：基线 + 草案

## 目标

隐式哈希锚点用于把自然剧本文本连接到工程系统：

- 本地化 Key。
- 存档位置。
- 热重载 Patch 定位。
- 编辑器诊断与源映射。
- 文本修订后的翻译对齐。

当前基线：

- 块级叙事节点使用显式节点名，服务跳转、图视图、调试、复入和存档结构定位。
- 行级文本使用隐式哈希锚点，服务本地化、文本修订对齐、热重载和源映射。
- 节点名和行级哈希不互相替代。节点名负责人类可读的图结构，哈希负责机器可追踪的文本身份。

## 第一版锚点规范

第一版实现使用 `line-v1` 锚点。输出格式为：

```text
l1_<fnv1a64-hex>
```

哈希输入：

```text
inscape-line-v1
kind=<Narration|Dialogue|ChoiceOption>
node=<node.name>
speaker=<speaker-or-empty>
occurrence=<same-content-index-within-node>
text=<normalized-text>
```

规范化规则：

- 统一换行符为 `\n`。
- 对文本执行 Unicode Form C 规范化。
- 去除首尾空白。
- 折叠连续空白为单个 ASCII 空格。
- 不做全角/半角标点转换。
- 不包含文件路径。
- 不包含绝对行号。

第一版取舍：

- 文件改名或移动不会改变锚点。
- 在同一节点内插入不同文本，不会改变已有文本锚点。
- 同一节点内出现完全相同的可翻译文本时，通过 occurrence index 区分。
- 如果在同一节点内同一文本前方插入相同文本，后续重复文本的 occurrence index 会改变；这类迁移后续需要显式稳定 ID 或提取工具辅助。
- 节点重命名会改变锚点。这是当前有意取舍，因为节点名仍是语义上下文的一部分。

## 候选锚点分层

为了同时满足唯一性和稳定性，可以考虑分层锚点：

- Text Hash：基于规范化原文，用于检测文本内容变化。
- Location Hash：基于文件路径与邻近结构，用于定位脚本位置。
- Semantic Anchor：基于显式标签、角色、分支路径或节点结构，用于迁移和对齐。
- Stable Override：作者或工具生成的显式稳定 ID，用于关键剧情节点。

是否采用分层设计待确认。

## 本地化流程草案

1. 扫描脚本并生成 AST。
2. 提取可翻译文本，例如对白、旁白、选项文本、UI 文案。
3. 为每条文本生成锚点、源位置、上下文和角色信息。
4. 输出 CSV 或 PO。
5. 后续扫描时对比旧目录，识别新增、删除、修改和可能移动的文本。
6. 对疑似文本微调的条目提供翻译继承建议，而不是直接丢弃旧译文。

当前已实现第一版 CSV 提取命令，详见 [本地化提取](l10n-extraction.md)。第一版提取范围包括旁白、对白、选择提示和选择项；metadata、演出标签、节点名和跳转目标暂不提取。

## 碰撞与版本策略

- IR 中的 `anchor` 字符串携带版本前缀 `l1_`。
- 哈希算法当前为 FNV-1a 64-bit，后续可替换为 XXHash/MurmurHash，但必须升级前缀。
- 编译器会检测非空 `anchor` 碰撞，并产生 `INS040` 错误诊断。
- 哈希算法升级必须提供迁移工具。

## 旧候选公式

早期材料中的候选公式为：

```text
ID = Hash(RawText + FilePath + ContextIndex)
```

它已被第一版 `line-v1` 替代。主要原因是 `FilePath` 和绝对行号会让文件移动、插入注释、插入 metadata 时产生不必要的锚点漂移。

## 待确认问题

- 本地化 Key 更应追求文本变化敏感，还是文本移动稳定。
- 译文对齐是否需要模糊匹配算法。
- 是否允许人工锁定某条文本的 Key。
- 对重复对白是否合并翻译，还是按上下文分别翻译。
- 存档锚点是否与本地化锚点共用，还是使用不同层级。
- 节点重命名是否需要迁移表，或是否允许对关键文本写显式稳定 ID。
