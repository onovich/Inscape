# 项目立项说明

状态：基线 + 草案

## 一句话定位

Inscape 是一套面向叙事驱动型游戏的 DSL、编译器、编辑器与运行时工具链，用自然剧本文本驱动确定性的游戏叙事执行。

## 项目愿景

创作者应该能把主要注意力留在文本、角色、选择与节奏上，而不是在 ID、配置表、跳转节点和资源调用之间来回切换。开发者则需要一套稳定、可调试、可自动化、本地化友好的数据链路。

Inscape 的核心价值在于连接这两端：

- 对创作者：提供接近剧本写作的 DSL。
- 对工具链：通过解析、哈希锚点、IR 与运行时管线把文本转成确定执行流。
- 对团队协作：让文本修订、本地化、存档、预览和调试可以被追踪、比较和自动化。

## 目标用户

- 视觉小说、互动小说、叙事 RPG、剧情向独立游戏创作者。
- 需要把大量文本、分支、变量、本地化和素材调度整合到同一流程中的小型团队。
- 希望在 Unity 内快速落地，但未来可能迁移到独立引擎的开发者。

## 核心目标

- 提供一套低噪声的剧本 DSL，让对话、标签、分支和变量更新可以在同一文本流中表达。
- 编译 DSL 到稳定 IR，供 Unity 或未来运行时执行。
- 使用隐式哈希锚点支持本地化 Key、文本修订对齐、存档恢复和热重载定位。
- 通过单向数据流管理叙事状态，使状态变更可追踪、可回放、可调试。
- 构建独立编辑器，支持语法高亮、大纲、错误提示、逻辑视图、实时预览和状态监视。

## 当前非目标

- 不在立项阶段设计完整游戏引擎。
- 不在 DSL 未定稿前承诺与 Ink、Yarn Spinner 或 Ren'Py 的完整兼容。
- 不把 AI 辅助创作作为早期核心依赖。
- 不在没有原型验证前锁定复杂的节点图编辑器交互。
- 不把所有 Unity 游戏逻辑都纳入 DSL，战斗、复杂 UI、演出系统应通过扩展边界接入。

## 第一阶段可交付范围

阶段 1 的目标是证明“文本到确定执行流”的主链路成立。

- 图叙事 DSL 草案与解析原型。
- 显式节点、对白、旁白、选项、跳转、注释和元信息的 IR 输出。
- 哈希锚点生成与增量提取原型。
- VSCode 语言支持原型，包含语法高亮、补全、诊断、跳转定义和大纲。
- HTML 调试预览原型，支持无引擎调试节点、选项、跳转和回环。
- Unity 宿主解释执行器原型，支持对话、背景、立绘切换与简单分支。
- 最小本地化导出流程，能从脚本生成 CSV 或 PO 形式的翻译表。

第一阶段暂不包含：

- 变量系统。
- 条件查询。
- 自定义指令。
- 复杂 Timeline 生成。
- 正式独立编辑器。

## 成功标准

- 一个短篇视觉小说片段可以只通过 Inscape DSL 驱动基础演出。
- 修改脚本文本后，可以重新生成 IR 与本地化表，并保留可解释的变更差异。
- 存档可以通过脚本锚点与状态快照恢复到确定位置。
- Unity 运行时不依赖散落的全局单例来推进剧本执行。
- 语法和工具链边界足够清晰，能够继续讨论编辑器和扩展系统。
- VSCode 内可以获得足够的提示和诊断，降低 DSL 记忆压力。
- HTML 预览可以在无 Unity 环境下验证叙事图。

## 主要风险

- 哈希锚点如果直接绑定原文，文本微调会造成翻译 Key 大量失效。
- 将文件路径和上下文序号纳入哈希可能提高唯一性，但也会降低重命名和移动文本时的稳定性。
- DSL 过于自然会降低解析确定性；过于工程化又会破坏写作心流。
- 实时热重载涉及运行中状态、当前指令栈和素材加载，必须先定义可安全刷新边界。
- 编辑器如果过早追求复杂可视化，可能拖慢核心编译器和运行时验证。
