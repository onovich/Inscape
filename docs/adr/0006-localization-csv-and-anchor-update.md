# 0006：第一版本地化使用 CSV 与锚点精确继承

状态：Accepted

## 背景

Inscape 的行级隐式 hash 需要尽快被真实工具链验证。第一阶段尚未进入 Unity 或独立编辑器，因此本地化流程必须能在 CLI 和 VSCode 阶段独立运行。

同时，翻译继承存在风险：如果过早做自动模糊匹配，可能把旧译文错误套到语义已经变化的新文本上。

## 决定

第一版本地化默认使用 CSV。

提供四个 CLI 命令：

- `extract-l10n`
- `extract-l10n-project`
- `update-l10n`
- `update-l10n-project`

`extract` 命令只输出当前脚本中的可翻译文本。`update` 命令读取旧 CSV，通过 `anchor` 精确继承 `translation`，并输出 `status`：

- `current`：当前仍存在并继承旧译文。
- `new`：当前新增。
- `removed`：旧表存在但当前不存在，保留供审校。

第一版不自动做模糊匹配。

## 原因

- CSV 易于检查、版本控制和交给外部翻译流程。
- 精确锚点继承确定性强，便于验证 hash 设计。
- 保留 `removed` 行可以减少误删译文，也方便后续迁移。
- 模糊匹配需要人工确认界面或报告格式，不应塞进第一版自动行为。

## 影响

- 文本轻微改写会被标记为 `new`，旧译文不会自动继承。
- 后续需要设计“疑似改写”候选列表，而不是直接覆盖译文。
- PO/XLIFF 可以作为后续导出格式，但不阻塞第一阶段。
- VSCode 和未来编辑器可以直接复用 CLI 命令实现一键导出和更新。

## 后续验证

- 在 VSCode 扩展中接入本地化导出/更新命令。
- 为模糊匹配设计人工确认报告。
- 通过真实剧本修改验证 `current/new/removed` 是否足够支持翻译协作。
